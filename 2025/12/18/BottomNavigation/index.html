整体架构原理：

该导航系统采用 MVVM + Navigation Component 架构，通过声明式的导航图管理 Fragment 之间的跳转关系

BottomNavigation的组成结构如下：

主要组成部分

1.导航图 (Navigation Graph)

文件：nav_graph.xml

定义了应用的所有导航目的地 (destinations)
startDestination 指定初始页面为 menu_home
每个 标签代表一个可导航的页面（部分代码中有search，profile；Dashboard，Notifications的名字错乱是套模板时留下的小bug只影响观看，不影响运行）

2.主 Activity 布局

文件：activity_main_bnav.xml 包含三个核心组件：
Toolbar - 应用栏，显示标题和操作按钮
NavHostFragment - Fragment 容器，承载导航内容
BottomNavigationView - 底部导航栏

3.底部菜单定义

文件：bottom_navigation_menu.xml

定义底部导航栏的菜单项，每个 item 的 id 需要与导航图中的 fragment id 保持一致。

4.主 Activity 控制器

文件：Main_bnav.kt 核心功能包括：

通过 findNavController 获取导航控制器
使用 AppBarConfiguration 配置应用栏行为
通过 setupWithNavController 将底部导航与 NavController 关联

5.Fragment 页面组件

以 HomeFragment 为例：
HomeFragment.kt - 视图逻辑控制器

HomeViewModel.kt - 数据模型

fragment_home.xml - 布局文件

项目文件总览：

工作流程

初始化阶段：

Main_bnav 加载布局，初始化 NavHostFragment
NavController 绑定到 BottomNavigationView
默认加载 menu_home 对应的 HomeFragment

导航过程：

用户点击底部菜单项
BottomNavigationView 通过 NavController 切换对应 Fragment
Toolbar 标题自动更新为当前 Fragment 的 label

这种设计实现了界面与导航逻辑的解耦，简化了页面跳转管理。

需要添加的依赖：

带有-ktx的是指在 navigation-fragment 的基础上增加了 Kotlin 扩展函数。提供了更流畅的 Kotlin 风格 API

如果你的项目是 Kotlin 为主，建议使用 navigation-fragment-ktx，因为它能显著提升开发效率和代码可读性。
如果你使用的是纯 Java 项目或不希望引入 KTX，可以只用 navigation-fragment。

📌 注意：navigation-fragment-ktx 依赖于 navigation-fragment，因此通常不需要单独声明两者，只需添加 ktx 版本即可自动包含基础库。

使用第三方库时的问题

Maven 仓库配置说明

配置语法解释
在 settings.gradle.kts 文件中配置 Maven 仓库时，需要使用正确的语法：
正确的配置方式

maven { url = uri(“https://www.jitpack.io“) }

语法要点

maven {}: 这是一个函数调用，接受一个 lambda 表达式
url = uri(): 使用 uri() 函数包装 URL 字符串，而不是直接使用字符串
uri(): Gradle 提供的函数，用于创建 URI 对象

配置说明

JitPack 仓库: https://www.jitpack.io 是一个常用的第三方库仓库
用途: 用于获取通过 GitHub 等平台发布的开源库
语法要求: 必须使用 uri() 函数包装 URL 字符串
